## Cross-Project Patterns

### Pattern: Scope Creep Is Universal
8/18 self-improvement sessions made the same watch.sh refactor regardless of task. This is not a Foreman-specific problem — it's an LLM autonomy problem. When given freedom, agents gravitate to the most obvious local improvement rather than the assigned task.

**Implication**: Constraint enforcement is more valuable than capability expansion. Adding file allowlists, output assertions, and scope locks would improve deliverable rates more than better prompts.

### Pattern: Activity ≠ Achievement
The harvester marks "success" based on session activity (output produced, no errors). This conflates running with completing. The 91.5% → ~56% gap is the signature of this conflation.

**Implication**: Every agent orchestration system needs deliverable verification, not just process monitoring. This is a publishable insight.

### Pattern: Rich Prompts Don't Help (Yet)
Exp 1: 100% both conditions. SWE-bench: 0% both conditions, but "full" actually performed WORSE (4/8 partial vs 7/9 partial for bare). Rich prompts make Claude do more work on the wrong things.

**Implication**: Prompt composition's value may be in skill selection and multi-session continuity, not individual task performance. The paper reframes this correctly.

## Product Signals

### Signal 1: "Deliverable Verification as a Service"
Every agent orchestrator needs to answer: "did the agent actually do what I asked?" Not "did it run" but "is the output file at the specified path with the right content?" This is a product.

**Who pays**: Anyone running autonomous agent workflows. CI/CD for agents.
**Implementation**: `foreman verify --deliverable research/paper/foreman-paper.md --check "contains 'Abstract' and len > 100 lines"`

### Signal 2: "Scope Lock for Agent Sessions"
A middleware that constrains which files an agent can modify. Like a sandbox for code changes.

**Who pays**: Enterprises running autonomous coding agents who can't have agents deleting directories.
**Implementation**: Git hook or file watcher that rejects diffs outside the allowlist.

### Signal 3: "Agent Session Viewer"
Watching 4 tmux sessions simultaneously requires deep terminal knowledge. A web-based session viewer with live output, task tracking, and outcome display would be 10x more accessible.

**Who pays**: Teams running multiple agent sessions. DevOps for AI.

### 2. Pre-Dispatch Validation Hook
Before spawning, check:
- Is prompt > 100 chars? (catches truncation bugs)
- Is workspace accessible?
- Is the skill appropriate for the task?
- Are there sibling sessions with overlapping scope?

### 3. Session Health Check Cron
Every 5 minutes:
- Is the session still authenticating? (OAuth stuck → kill + retry)
- Has it been thinking > 10 min with no tool calls? (infinite think → interrupt)
- Has it modified files outside its scope? (scope creep → flag)
