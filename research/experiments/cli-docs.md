# Foreman CLI Reference

The `foreman` CLI is a thin Bash wrapper over the Foreman HTTP API (`http://127.0.0.1:7374`). Every command maps to one API call. Override the service URL with `FOREMAN_URL`.

## Commands

### `foreman status`

Portfolio overview: all goals, active sessions (with live tmux state), recent decisions, and events.

```bash
# Full portfolio snapshot
foreman status
```

**API:** `GET /api/status`

**Returns:** `{ goals, sessions, recentDecisions, events }` where each session is enriched with `alive` (tmux process exists), `idle` (no output recently), and `lastOutput` (last line from tmux pane).

---

### `foreman goals`

List all active goals. Goals are the top-level unit of work ÔÇö they span domains, not repos.

```bash
# List all goals
foreman goals
```

**API:** `GET /api/goals`

**Returns:** Array of `{ id, intent, status, workspace_path, workspace_type, context, priority, created_at, updated_at }`.

### `foreman goal <intent>`

Create a new goal from a free-text intent string.

```bash
# Create a goal
foreman goal "Drive phony voice cloning to SOTA"

# Multi-word intents are concatenated
foreman goal Ship the GTM launch for sandbox blueprints
```

**API:** `POST /api/goals` with `{ "intent": "<all args joined>" }`

**Returns:** The created goal object.

---

### `foreman dispatch <task>`

Dispatch a task to a Claude Code session. Foreman creates an isolated git worktree, composes a prompt (project context, learned flows, dead ends, operator taste), and spawns a tmux session.

```bash
# Simple dispatch ÔÇö Foreman picks the workspace from the active goal
foreman dispatch "Fix the auth middleware token storage"

# With a specific skill
foreman dispatch --skill /evolve "Improve voice cloning MOS score"

# With a specific workspace
foreman dispatch --workspace ~/code/phony "Add streaming endpoint"

# Skill + workspace
foreman dispatch --skill /pursue --workspace ~/code/foreman "Build the planning layer"
```

**Options:**
- `--skill <name>` ÔÇö Slash-command skill to invoke (e.g., `/evolve`, `/pursue`, `/polish`). Affects prompt composition, model selection, and confidence tracking.
- `--workspace <path>` ÔÇö Working directory. Defaults to the goal's `workspace_path`.

**API:** `POST /api/dispatch` with `{ "task", "skill", "workspace" }`

**Returns:** `{ decision, session, backend, model, worktree, branch, baseBranch, confidenceLevel, confidenceScore, promptLength, promptTier, promptSections, promptPreview }`

The session runs in an isolated worktree branched from the operator's current branch. When done, it commits and pushes to a PR for operator review. Sessions never touch the operator's working directory.

---

### `foreman sessions`

List all sessions (active and historical) with live tmux state.

```bash
# List all sessions
foreman sessions
```

**API:** `GET /api/sessions`

**Returns:** Array of `{ name, goal_id, decision_id, work_dir, status, started_at, last_checked_at, alive, idle }`.

Session statuses: `starting` Ôćĺ `running` Ôćĺ `idle` Ôćĺ `success`/`failure`/`dead`.

`alive` indicates whether the tmux session process still exists. `idle` indicates whether the session has produced output recently.

---

### `foreman watch`

Open a live tmux grid showing all active Foreman sessions. Each pane streams one session's output in real time. Supports up to 6 sessions in a tiled layout.

```bash
# Opens an interactive tmux session ÔÇö press Ctrl+B then D to detach
foreman watch
```

This is not an API call. It runs `scripts/watch.sh` which:
1. Finds all `foreman-*` tmux sessions
2. Pipes each session's output to a temp file via `tmux pipe-pane`
3. Creates a new `foreman-watch` tmux session with one pane per active session
4. Attaches interactively

---

### `foreman decisions [N]`

Show the last N decisions (default: 10). Decisions are the unit of dispatch ÔÇö each one records what was dispatched, why, and what happened.

```bash
# Last 10 decisions
foreman decisions

# Last 50 decisions
foreman decisions 50
```

**API:** `GET /api/decisions?limit=N`

**Returns:** Array of `{ id, goal_id, skill, task, reasoning, status, origin, outcome, learnings, metrics, taste_signal, session_name, worktree_path, worktree_branch, base_branch, template_version, cost_usd, deliverable_path, deliverable_spec, scope_spec, prompt_sections, deliverable_status, created_at, updated_at }`.

Decision statuses: `dispatched` Ôćĺ `success`/`failure`.

The `origin` field distinguishes `operator` dispatches (via CLI/Pi) from `auto` dispatches (generated by the planning layer or self-improvement loop).

---

### `foreman plans`

List strategic plans. Plans are generated by dispatching an ideation session that analyzes goals, recent decisions, learned flows, and taste signals to propose next actions.

```bash
# List all plans
foreman plans
```

**API:** `GET /api/plans`

**Returns:** Array of plan objects with status, rationale, and proposed actions.

### `foreman plan`

Generate new strategic plans. Dispatches an ideation session that reads the full portfolio state and produces ranked plan proposals.

```bash
# Generate new plans (async ÔÇö spawns a session)
foreman plan
```

**API:** `POST /api/plans/generate`

The ideation session receives: active goals, recent decisions (last 15 successes/failures), learned flows, skill preferences, taste signals, dead ends, recent projects, and aggregate stats. It writes structured plan output to the worktree.

---

### `foreman confidence`

Show confidence levels for each (skill, project) pair. Confidence determines autonomy: low-confidence actions require operator approval, high-confidence actions run autonomously.

```bash
# Show all confidence entries
foreman confidence
```

**API:** `GET /api/confidence`

**Returns:** Array of `{ skill, project, score, level, signals, override }`.

Confidence levels map to autonomy tiers:
- **0.0ÔÇô0.3** ÔÇö Dry-run only
- **0.3ÔÇô0.6** ÔÇö Propose and wait for approval
- **0.6ÔÇô0.8** ÔÇö Act and notify operator
- **0.8ÔÇô1.0** ÔÇö Fully autonomous

Confidence updates on every outcome: `agree` (operator approves) increases it, `disagree` (operator rejects) decreases it. The operator can set manual overrides per project.

## Additional Commands

### `foreman learn`

Trigger the learning loop manually. Scans operator sessions (Claude/Pi/Codex), extracts workflows, skill preferences, and dead ends, and stores them as learnings.

```bash
foreman learn
```

**API:** `POST /api/learn`

### `foreman logs`

Follow the Foreman systemd service logs in real time.

```bash
# Live log stream ÔÇö Ctrl+C to stop
foreman logs
```

Uses `journalctl --user -u foreman -f`.

### `foreman restart`

Restart the Foreman systemd service.

```bash
foreman restart
```

### `foreman update`

Pull the latest code and restart the service.

```bash
foreman update
```

Runs `git pull --rebase` in `$FOREMAN_HOME/repo` (default `~/.foreman/repo`), then restarts the service.

## Environment Variables

| Variable | Default | Description |
|---|---|---|
| `FOREMAN_URL` | `http://127.0.0.1:7374` | Service URL for CLI requests |
| `FOREMAN_PORT` | `7374` | Port the service listens on |
| `FOREMAN_HOME` | `~/.foreman` | Data directory (DB, logs, worktrees) |
| `FOREMAN_MAX_DAILY_COST` | `20` | Daily cost cap in USD |
| `FOREMAN_MAX_SESSIONS` | `5` | Max concurrent tmux sessions |
| `FOREMAN_AUTO_MERGE` | `false` | Auto-merge PRs (default: operator reviews) |
| `CLAUDE_PATH` | `~/.local/bin/claude` | Path to Claude CLI binary |
